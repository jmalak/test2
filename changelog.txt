Change 38191 on 2020/12/11 by MichalN@MICHALN_LINUX_64VM_RYZEN

	ChangeList: The declaration of execv(), execve(), and execvp() in the
	Linux runtime libraries has been adjusted to match POSIX 1003.1 and
	later specifications. POSIX declares (in unistd.h) the argv and envp
	arguments as arrays of const pointers to non-const strings, while DOS,
	Windows, and OS/2 declare them (in process.h, optionally also unistd.h)
	as arrays of const pointers to const strings.
	
	Implementor's Note: This is an annoying discrepancy which has its origin
	in distant history. Starting with Microsoft C 6.0 in 1989, Microsoft
	defined argv/envp as const pointers to const strings. UNIX systems
	starting circa 1990 defined them as const pointers to non-const strings.
	The POSIX standard explicitly notes that the double const would be in
	fact logical but would "disallow existing correct code".
	 The issue is purely a syntactic problem because if the exec functions
	succeed, there is no caller left to worry about the strings being
	modified, and if they faili, they indeed do not modify the memory.
	 It would be nice to keep process.h and unistd.h consistent across
	platforms, but clashing declarations in process.h vs. unistd.h would
	be worse than platform differences in the execv family of functions.
	
	This changeset modifies both the library and several instances where
	using the POSIX declarations in fact makes casts redundant in Linux
	specific code.
